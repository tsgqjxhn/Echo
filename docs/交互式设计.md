# 交互式设计

本文档用于统一 Echo 剧情中的互动事件、素材生成、前端触发、游戏嵌入和后续研发字段。目标是让 `docs/剧情.md` 里的剧情正文保持可读，同时让图片、朋友圈、H5、头像、死亡、分支、线索等内容都有稳定标记，方便后续生成、导入、编程和检查。

## 指导类

这一类只放实现规则、字段说明、兼容写法、检查顺序和落地方法；它不直接代表一条具体要生成或开发的内容。

### 全局规则

#### 消息显示控制标记

这些标记用于控制聊天界面的表现，不一定改变剧情，但会影响玩家感受到的节奏、真实感和压力。

| 标记 | 用途 | 是否建议前端识别 | 说明 |
|---|---|---:|---|
| `[隐藏系统消息]` | 只让系统执行效果，不显示给玩家 | 是 | 适合头像、名字、素材解锁、状态变化等后台事件 |
| `[对方正在输入]` | 显示输入中状态 | 是 | 用于紧张、犹豫、长消息前的停顿 |
| `[消息延迟]` | 控制下一条消息延迟出现 | 是 | 可带秒数，如 `[消息延迟:8s]` |
| `[消息撤回]` | 模拟对方撤回一句话 | 可选 | 适合制造信息缺口和不安感 |
| `[消息不可见]` | 记录剧情发生但本条不进入聊天流 | 是 | 与隐藏系统消息相似，但强调该消息本身不展示 |
| `[未读数更新]` | 调整会话列表未读提示 | 可选 | 适合多会话或假离线状态 |
| `[联系人上线]` | 联系人状态变为在线 | 可选 | 适合断联后恢复 |
| `[联系人离线]` | 联系人状态变为离线 | 可选 | 适合死亡错路、信号丢失、潜入阶段 |

建议写法：

```text
【系统】[隐藏系统消息][消息延迟:6s][对方正在输入]
```

#### 统一事件字段

所有需要生成、设计、编程或检查的互动事件，建议统一拆成以下字段。剧情文档里可以保留简写标记，专门的交互文档或数据表里使用完整字段。

| 字段 | 必填 | 说明 |
|---|---:|---|
| `id` | 是 | 全局唯一事件 ID，如 `img-d03-park-light-01` |
| `type` | 是 | 事件类型，如 `image_insert`、`avatar_update`、`h5_game` |
| `day` | 是 | 第几天，60 天版本只允许 `1-60` |
| `volume` | 建议 | 所属卷，便于批量检查节奏 |
| `scene` | 是 | 所属场景，如雨夜公园、旧校门房、灯塔外环 |
| `trigger` | 是 | 触发时机，写明“在哪条消息后触发” |
| `target` | 建议 | 影响对象，如星、玩家、朋友圈、图鉴、小游戏 |
| `payload` | 是 | 事件具体内容，图片标题、动态文案、游戏名称等 |
| `visible` | 是 | 是否对玩家可见 |
| `blocking` | 是 | 是否阻塞剧情继续，H5 和关键选择通常为 `true` |
| `rollback` | 是 | 死亡或失败后是否允许回溯 |
| `effects` | 建议 | 对剧情状态、角色状态、素材、朋友圈、分支的影响 |
| `assetId` | 可选 | 关联图片、音频、视频、头像、背景等资源 ID |
| `priority` | 可选 | 事件优先级，多个系统事件同一时刻触发时排序 |
| `status` | 建议 | `todo`、`designed`、`generated`、`implemented`、`verified` |
| `notes` | 可选 | 给编剧、画图、前端、测试的备注 |

统一模板：

```yaml
- id:
  type:
  day:
  volume:
  scene:
  trigger:
  target:
  payload:
  visible:
  blocking:
  rollback:
  effects:
  assetId:
  priority:
  status:
  notes:
```

#### 当前代码可识别的系统写法

为了兼容现有前端，短期内建议继续保留这些写法。后续如果改成统一方括号标记，需要同步更新导入脚本和前端识别逻辑。

##### 4.1 名字更新

当前可识别：

```text
【系统】联系人名称更新为【星】。
【系统】联系人名称由【未知用户】永久更新为【星】。
【系统】联系人暂记为：星
```

建议标准标记：

```text
【系统】[名字更改:未知用户=>星][隐藏系统消息]
```

完整字段：

```yaml
- id: name-d02-xing
  type: name_update
  day: 2
  scene: 雨夜后续对话
  trigger: 星第一次正式说出名字后
  target: conversation.contactName
  payload:
    from: 未知用户
    to: 星
  visible: false
  blocking: false
  rollback: false
  effects:
    - 更新聊天顶部联系人名称
  status: implemented-compatible
```

### 分类字段与规范

#### 图片插入设计规范

图片插入承担四种作用：推进主线、制造误导、补充环境、二次解谜。不要把图片只当插画，最好每张都能回答一个剧情问题，或制造一个新的问题。

##### 6.1 当前兼容写法

当前前端会识别类似格式：

```text
【系统】[图片插入：剧院高空俯拍——对应当前线索的补充画面]
```

识别逻辑会拿 `剧院高空俯拍` 去匹配图鉴标题，匹配成功后自动插入真实图片消息：

```text
[IMAGE:<asset-src>]
```

##### 6.2 建议标准标记

```text
【系统】[图片插入:scene-theatre|剧院高空俯拍|证据图][图鉴解锁:scene-theatre][隐藏系统消息]
```

#### 朋友圈发布设计规范

朋友圈负责承接星的情绪、制造“她在另一个生活界面里仍然存在”的真实感，也可以作为伏笔和误导。

##### 7.1 当前兼容写法

```text
【系统】朋友圈动态插入：今天先别死。
```

当前前端会提取冒号后的内容，生成一条 moments post 并弹出提示。

##### 7.2 建议标准标记

```text
【系统】[朋友圈发布:moment-d08-01][隐藏系统消息]
```

##### 7.4 朋友圈内容原则

| 原则 | 说明 |
|---|---|
| 不解释主线 | 朋友圈不应替代主线对话，只做侧面证据 |
| 可以嘴硬 | 星的动态可以比聊天更轻、更冷、更像真实社交平台 |
| 可以误导 | 允许看似日常，回看时才发现是求救或定位 |
| 控制频率 | 每卷建议 2-4 条，关键日可以额外加一条 |
| 允许图片 | 图片型朋友圈需要同时登记 `imageId` |

#### H5 / 小游戏设计规范

H5 和小游戏不是装饰，它应该承担“玩家必须亲手参与一次”的剧情压力。每个游戏都要明确规则、玩法、成功条件、失败后果和剧情影响。

##### 8.1 当前兼容写法

```text
【系统】H5小游戏插入：旧馆门缝撬动节奏
```

当前前端检测到 `H5小游戏插入` 后会弹出已有 puzzle 游戏。

##### 8.2 建议标准标记

```text
【系统】[H5嵌入:game-d06-door-rhythm][小游戏开始][隐藏系统消息]
```

##### 8.4 游戏玩法概要类型

| 类型 | 适用场景 | 玩法概要 | 编程重点 |
|---|---|---|---|
| `puzzle` 解谜 | 手册、档案、密码 | 拖拽、排序、拼接、选择正确组合 | 状态校验、错误反馈 |
| `escape` 逃生 | 追踪、巡逻、封锁 | 选择路线、管理风险、触发成功率 | 路线状态、失败回溯 |
| `route` 路线规划 | 校区、港区、灯塔 | 在地图节点中选择路径 | 节点图、可达性、危险等级 |
| `timing` 节奏操作 | 门缝、配电箱、巡检节拍 | 在正确时机点击或停顿 | 计时器、容错窗口 |
| `decode` 解码 | 广播、报码、档案编号 | 对照表、替换、反向读取 | 输入校验、提示系统 |
| `compare` 图片比对 | 监控、照片、截图 | 找不同、叠图、放大线索 | 图片交互、区域命中 |
| `stealth` 潜行 | 旧校、港区、外环 | 避开巡逻视线或时间段 | 敌方路径、失败条件 |
| `social_disguise` 社交伪装 | 群聊、论坛、朋友圈 | 选择合适话术维持身份 | 对话分支、信任值变化 |

#### 死亡事件与回溯设计

死亡事件必须结构化，否则玩家不知道为什么死，前端也难以决定是否进入 `dead` 状态。

##### 9.1 死亡类型

| 标记 | 含义 | 示例 |
|---|---|---|
| `[死亡事件:直接死亡]` | 星生理死亡、爆炸、坠落、被处决等硬失败 | 被抓住、救生机爆炸 |
| `[死亡事件:软死亡]` | 没死，但被洗脑、永久失联、失去行动能力 | 脑电波被抹平、记忆被覆盖 |
| `[死亡事件:关系死亡]` | 星不再信任玩家，连接断裂 | 玩家持续诱导错误选择 |

#### 汇总统计与落地口径

- 图片事件：91 个，均使用 `[图片插入:id|标题|用途]` 并同步 `[图鉴解锁:id]`。
- H5/小游戏事件：58 个，均绑定 `[选择出现]` 与 `[回溯点]`。

- 语音/音频/视频：22 个，其中语音 2 个、音频 14 个、视频/动态 6 个。
- UI/状态事件：32 个；线索事件：12 个；每日错路回溯入口：60 个。

#### 需要研究 / 策划的问题

1. 60 天剧情中哪些节点必须有图片，哪些只需要文字。
2. 每卷 H5 数量是否固定，建议每卷 1-2 个，不宜过多。
3. 死亡回溯是否回到当天选择点，还是回到上一条安全消息。
4. 星的信任值是否真实影响剧情，还是只作为编剧标记。
5. 朋友圈是否允许玩家互动，如点赞、评论、收藏。
6. 图鉴是否只是收藏，还是能反向解锁隐藏文本。
7. H5 失败是否允许继续主线，还是必须成功。
8. 图片生成风格是否全局统一，还是按卷调整色彩和镜头。
9. 音频和视频是否进入第一版，还是后续版本再做。
10. 所有系统标记是否最终从 Markdown 导入为结构化 JSON。

#### 推荐落地顺序

1. 先统一 `docs/设计.md` 的系统标记格式。
2. 再整理 60 天图片清单，补齐每张图的生成字段。
3. 接着整理名字/头像切换事件，保证第 2、15、46 天不丢。
4. 然后整理朋友圈发布清单，按卷分配情绪节奏。
5. 再做 H5/小游戏清单，先选每卷最关键的 1 个。
6. 最后补死亡、回溯、线索、角色状态字段。
7. 等文档稳定后，再同步修改导入脚本和前端识别逻辑。

#### 导入与开发检查清单

1. 先把本节图片、H5、媒体、朋友圈、线索拆成结构化 JSON，字段沿用前文统一事件字段。
2. 图片资源先生成占位 assetPath，即使真实图片未出，也要保证聊天附件和图鉴不会空白。
3. H5 第一版优先实现通用容器：`route`、`compare`、`puzzle`、`decode/timing` 四类，具体关卡通过配置驱动。
4. 语音/音频/视频必须有 `transcript`，并在聊天里提供静音可读版本。
5. 隐藏系统消息进入事件日志，但不进入普通玩家聊天气泡；导入脚本要校验每个 `id` 全局唯一。
6. 回归测试至少覆盖：名字变更第 2 天、头像第 2/15/46 天、图片插入与图鉴解锁、H5 成功/失败、朋友圈发布、死亡回溯、刷新后状态保持。
