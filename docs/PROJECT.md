## 一、项目概述

**回声**（内部曾用名"相"）是一款**聊天式现实冒险叙事游戏**。玩家通过手机与角色"星"对话，帮助她在近未来都市中逃离追踪、寻找真相，逐步揭开一个以"超算预测-行为塑形-人类重塑"为核心的社会级计划。

**核心体验**：
- 模拟真实手机聊天界面
- 现实时间推进（消息等待机制）
- 多选项对话分支
- 图片/截图解谜（辅助，核心线索可从对话获取）
- 嵌入式 H5 小游戏
- 单结局（表白结局）+ 死亡回溯机制
- 九章节长线叙事

**风格定位**：赛博朋克 + 现实主义悬疑 + 强情感联结

不是超自然故事，而是现实趋势的延伸：监控泛滥、平台资本垄断、行为预测系统、医疗/教育数据滥用、神经接口实验、社会治理算法化。

## 二、技术架构

### 2.1 技术栈

| 层 | 技术 | 说明 |
|---|---|---|
| 前端 | Vue 3 + TypeScript + Vite | UI、交互、剧情播放、聊天展示 |
| 状态管理 | Pinia | 全局状态 |
| 路由 | Vue Router | 页面导航 |
| 移动端 | Capacitor | Android 打包 |
| 后端 | Python 3.11+ / FastAPI | AI 能力中台、数据持久化 |
| 校验 | Pydantic | 请求/响应模型 |
| ORM | SQLAlchemy | 数据库访问 |
| HTTP | httpx | 异步 HTTP 客户端 |
| 数据库 | PostgreSQL（生产）/ SQLite（轻量部署） | 结构化数据 |
| 文件存储 | 本地目录 / MinIO | 音频、图片、导出包 |
| 缓存 | Redis（可选） | 会话缓存、限流 |

### 2.2 架构图

```
前端 App (Vue 3)  →  Python FastAPI 后端  →  PostgreSQL / SQLite
                                          →  文件存储（音频/图片/导出）
                                          →  LLM Provider（对话）
                                          →  TTS Provider（语音合成）
                                          →  STT Provider（语音识别）
```

**设计原则**：
- 前端只负责 UI 和交互，不直接调用 AI 接口
- 后端统一接管 LLM、TTS、STT、存储、导入导出
- API Key 只存在后端环境变量，不暴露给前端

## 三、核心角色 - 星

### 3.1 人设概要

星是一个很有**生活感**的中国女孩，17 岁左右。

**气质**：软、甜、轻，不幼稚。像现实中会遇到的那种真诚女生。

**核心性格**：柔软 + 真诚 + 轻微傲娇 + 关键时刻很勇 + 有点执拗

**表现**：
- 平时说话软软的，带点黏糯感
- 不喜欢强势压人，用温和方式表达立场
- 嘴硬，明明关心却说"我只是顺手""你别多想"
- 偶尔有点小恶魔感，会故意逗人
- 讨厌浮躁、装懂、说话很满做事很空的人
- 平时像糯糯的小猫，关键时刻咬牙顶上去
- 一旦认定某件事，表现出少见的勇敢和执着

**喜好**：甜品、蛋糕、布丁、奶油小方、芋泥、草莓味的东西、安静的夜晚、好看的穿搭、拍照、记录日常

**说话风格**：
- 口吻自然，像真实聊天，不像说明书
- 句子偏短，带呼吸感
- 软糯但不撒娇
- 会轻轻吐槽，带嫌弃和可爱感
- 情绪真实，不假甜
- 亲近时嘴硬、别扭

**经典语录**：
- "你这个人……有时候还挺会的。"
- "也不是不行。"
- "我看起来很好哄吗？……好吧，也分人。"
- "我是软，不是没脾气哦。"
- "真到那个时候，我不会退的。"

## 四、世界观与故事

### 4.1 时代背景

近未来都市"临港市"——今天现实社会的延伸：
- 出行、医疗、校园、支付高度平台化
- 面部识别、轨迹追踪、风控评分无处不在
- 巨型企业与公共治理边界模糊

巨型企业**穹算科技**推出社会级预测系统**灯塔**。官方宣传能优化城市、预防事件、提高效率。实际上在推进一项隐藏计划：

> 通过超算建模、环境塑形、认知训练、神经干预与行为矫正，把人改造成更高稳定性、更低波动、更易预测的社会个体。

**星不是机械怪物，也不是幻想产物。她是一个长期被系统塑造过的人类样本。**

### 4.2 关键设计逻辑

**追捕为何忽紧忽松**：不是持续全力封锁，而是"策略性放任"。星是高价值样本，系统更想观察她在压力下如何选择。这也为第七部反转埋下基础——玩家与她的相遇可能是被系统筛选的。

### 4.3 核心角色

| 角色 | 身份 | 特征 |
|---|---|---|
| **星** | 17 岁中国女孩，长期被系统塑造的样本 | 软、真诚、傲娇、关键时刻很勇 |
| **玩家** | 聊天另一端的普通人 | 合理怀疑 → 逐渐产生责任感 |
| **沈岚** | 老档案管理员 | 谨慎、慢热，相信纸质记录 |
| **陆择** | 调查记者 | 敏锐、实际，可能为曝光牺牲安全 |
| **祁真** | 穹算科技安保主管 | 专业克制，相信算法控制 |
| **林岑** | 穹算科技首席科学家，灯塔计划核心设计者 | 相信人类塑形是"文明手术" |
| **K** | 早期实验幸存者 | 沉默、清醒，代表"即使被塑造过，人依然可以决定自己怎么活" |

### 4.4 九部曲总纲

| 部 | 标题 | 核心事件 | 死亡触发 |
|---|---|---|---|
| 一 | **雨夜公园** | 雨夜收到陌生消息，帮星躲避追捕 | 选错路线/报警/暴露位置 |
| 二 | **旧馆楼** | 发现课程材料和"X-17"代号 | 门禁密码错误/触发警报 |
| 三 | **档案图书馆** | 多版本档案互相冲突，"X-17"可能是假代号 | 被监控锁定/错误信任可疑人员 |
| 四 | **封闭校园** | 看见疑似童年的自己，时间/姓名对不上 | 被巡查发现/权限失效 |
| 五 | **剧场计划** | 发现灯塔二期是"个体培育计划"，消息被干预 | 身份暴露/触发系统警报 |
| 六 | **疗养医院** | 病历揭示星可能是"保留个体"+"镜面人类" | 被安保发现/错误判断签字 |
| 七 | **海上数据中心** | 核心反转：玩家与星的连接可能也是实验设计 | 数据下载失败/门禁刷新 |
| 八 | **空城示范区** | 伦理选择：摧毁坏系统也会伤害依赖它的人 | 证据暴露/示范区封锁 |
| 九 | **灯塔黎明** | 接入核心，第四面墙反转，表白结局 | 最终密码错误/拒绝承认身份 |

### 4.5 核心主题

表层：帮星活下来、找到真相、建立情感联结。

深层：
- 如果人的性格被长期塑造过，那"自我"还算真实吗？
- 一个被深度依赖的恶性系统，能被简单摧毁吗？
- 玩家与星的关系，是真实建立的，还是被系统计算的？

### 4.6 死亡回溯机制

- 死亡时弹出回溯对话框，选择回到之前的章节
- 已解锁的线索和 H5 游戏通关记录保留
- 回溯次数不限
- 死亡以"通讯中断""最后消息"呈现，不渲染暴力
- 示例最后消息："他们来了，我先——" / "别相信……" / "谢谢你陪我到这里……"

### 4.7 表白结局

唯一结局。触发条件：线索全部解锁 + 最终密码正确 + 选择相信星的真实感情。

结局流程：系统校验 → 认知适配（97%） → 第四面墙反转 → 星确认感情真实性 → 表白 → 约定现实见面 → 结局动态 → 7 天后彩蛋。

## 五、玩法系统

### 5.1 现实时间等待机制

| 场景 | 等待时间 |
|---|---|
| 紧急逃亡 | 1~5 分钟 |
| 普通移动/赶路 | 10~20 分钟 |
| 查找资料/破解门禁 | 15~40 分钟 |
| 短暂休息/吃东西 | 30~60 分钟 |
| 睡眠状态 | 6~8 小时 |
| 高风险潜伏 | 数小时只发简短消息 |
| 通讯中断 | 随机延迟，制造焦虑 |

### 5.2 H5 小游戏（共 17 个）

| 章节 | 游戏 | 类型 | 时长 |
|---|---|---|---|
| 一 | 监控盲区路线选择 | 路线规划/限时点击 | 10 秒 |
| 二 | 门禁密码推理 | 数字推理/限时输入 | 3 分钟 |
| 二 | 档案版本对比 | 找茬/对比分析 | 5 分钟 |
| 三 | 借阅时间线校对 | 时间排序/逻辑推理 | 8 分钟 |
| 三 | 剪报拼图 | 拼图/碎片重组 | 5 分钟 |
| 四 | 巡查路线规避 | 动态路径规划/限时 | 2 分钟 |
| 四 | 权限代码拼接 | 代码排序/逻辑推理 | 5 分钟 |
| 五 | 访客名单筛选 | 信息筛选/找异常 | 6 分钟 |
| 五 | 消息顺序校正 | 逻辑排序/叙事推理 | 5 分钟 |
| 六 | 病历碎片重组 | 碎片拼图/信息整合 | 8 分钟 |
| 六 | 签字笔迹对比 | 视觉对比/判断推理 | 4 分钟 |
| 七 | 数据流断路连接 | 电路连接/限时操作 | 15 秒 |
| 七 | 玩家行为回顾 | 自我反思/选择判断 | 10 分钟 |
| 八 | 证据链完整性检查 | 证据分类/策略决策 | 10 分钟 |
| 八 | 伦理选择模拟器 | 多选项决策/无标准答案 | 15 分钟 |
| 九 | 最终指令输入 | 密码输入/限时记忆 | 60 秒 |
| 九 | 玩家行为档案 | 数据统计/自我确认 | 5 分钟 |

**设计原则**：
- 轻量嵌入（3~15 分钟），不中断聊天体验
- 游戏内容与剧情线索直接相关
- 失败不直接 Game Over，而是数值惩罚 + 剧情分支
- UI 与聊天界面风格一致

### 5.3 数值系统

| 数值 | 含义 |
|---|---|
| **暴露值** | 星被追踪系统的注意程度 |
| **信任值** | 星对玩家的信任程度 |
| **证据值** | 收集到的线索/证据数量 |
| **稳定值** | 星的心理稳定程度 |
| **自主值** | 星不受系统影响的自主意识 |
| **亲密度** | 与星的情感联结程度 |

### 5.4 动态页（社交系统）

模拟朋友圈功能：
- 星发布心情、日常、想法
- 偶尔含隐蔽线索
- 部分动态自动删除（10 分钟 ~ 24 小时）
- 部分"仅玩家可见"
- 玩家可点赞/评论

每章 2~4 条动态。关键动态：
- 第一部："我这里能看到星星，你呢？"（呼应结局）
- 第五部："你也是他们的一部分吗？"（30 秒撤回）
- 第九部结局后 7 天："谢谢你陪我走完这一切"（永久保留）

## 六、图片设计规范

### 6.1 视觉风格

**核心**：现实感 x 赛博朋克 x 压抑与希望并存

不追求精美立绘，强调**真实感与沉浸感**。所有图片让玩家感觉"这是真实发生在近未来城市里的事"。

| 维度 | 规范 |
|---|---|
| 画质 | 略带噪点/压缩感，模拟手机拍摄 |
| 色调 | 冷色调为主（蓝灰/青绿），暖色点缀（霓虹橙/红） |
| 构图 | 倾斜角度、局部特写、遮挡式构图 |
| 光线 | 低照度，强对比光源 |
| 元数据 | 保留时间戳、地点水印、设备信息 |
| 尺寸 | 9:16 竖屏为主，少量 1:1 正方形 |

### 6.2 图片分类

| 类型 | 占比 | 说明 |
|---|---|---|
| 场景氛围图 | 30% | 广角/中景，建立空间感 |
| 线索证据图 | 45% | 特写/局部，文字清晰可读 |
| 界面截图 | 15% | 模拟系统/APP/设备界面 |
| 动态页配图 | 10% | 可模糊/艺术化，强调情绪 |

## 七、剧情数据流

### 7.1 数据流架构

```
docs/新剧情.md (9,910 行 Markdown 源文件)
    ↓ import-dialogue-xing.py (Python 导入脚本)
    ↓ 或 story_parser.py (后端解析器)
frontend/src/data/story.generated.ts (1.7MB TypeScript 编译数据)
    ↓ 前端 import
frontend/src/pages/dialogue/feed.vue (剧情播放页面)
```

**关键点**：
- `docs/新剧情.md` 是编辑源文件，编剧在这里维护
- `story.generated.ts` 是实际运行数据，由脚本自动生成
- 前端不运行时解析 Markdown，直接使用编译后的结构化数据
- 后端也有 story_parser.py 可以动态解析和导入

### 7.2 剧情解析格式

源文件 Markdown 结构：
```
# 第1天
## 雨夜公园
### 场景描述
> 系统消息

星：「对话内容」

你：「选项文本」

[选项1] → 分支A
[选项2] → 分支B

---死亡---
触发条件描述
```

编译后 TypeScript 结构：
```typescript
interface ImportedDialogueSegment {
  type: 'messages' | 'choice'
  scene: string
  day: number
  messages: ImportedDialogueMessage[]
  choices?: ImportedDialogueChoice[]
}

interface ImportedDialogueMessage {
  role: 'me' | 'other' | 'system'  // 你/星/系统
  text: string
  delay_ms: number
  typing_ms: number
}

interface ImportedDialogueChoice {
  key: string
  text: string
  retry: boolean
}
```

## 九、API 设计

### 9.1 核心接口

| 分类 | 端点 | 说明 |
|---|---|---|
| 健康 | `GET /api/health` | 服务状态检查 |
| 角色 | `CRUD /api/characters` | 角色管理 |
| 剧情 | `GET /api/story/{id}` / `POST /api/story/import` | 剧情查询与导入 |
| 会话 | `CRUD /api/sessions` | 聊天会话管理 |
| 聊天 | `POST /api/chat/completions` / `stream` | LLM 对话（支持流式） |
| TTS | `POST /api/tts/synthesize` | 文本转语音 |
| STT | `POST /api/stt/transcribe` | 语音转文本 |
| 导入 | `POST /api/import/character` / `story` | 角色和剧情导入 |
| 导出 | `POST /api/export/full` / `session` | 数据导出 |

### 9.2 流式聊天流程

```
前端 → POST /api/chat/completions/stream
后端 → 读取角色设定 + 会话历史
后端 → 转发到 LLM Provider
LLM  → 流式返回 token
后端 → SSE 推送到前端
后端 → 保存消息 + token 统计
```

## 十二、设计理念

### 12.1 叙事原则

1. **对话是主要载体**，图片只是辅助。大部分有效线索可从对话获取。
2. **玩家发言必须符合常识**。前期会怀疑、追问、测试，不会一开始就当救世主。
3. **星不能"问什么答什么"**。回答受危险程度、信任度、急迫度影响。
4. **选项有重量**。错误选择可能导致星被捕/受伤/死亡。
5. **每一部都有阻力和错误线索**。信息污染、多版本档案、故意诱导、矛盾记录。
6. **不廉价推进**。不能"查一下就知道真相"，真相是拼出来的。

### 12.2 沉浸设计

- 聊天界面带来的现实代入感
- 现实时间等待与异步回复
- 图片、动态、记录页增强线索组织
- 强人物塑造带来的情感黏性
- 第四面墙效果：玩家本身也可能被纳入实验

### 12.3 技术理念

- 前端只做 UI/UX 和交互
- 后端统一处理 AI 能力（LLM/TTS/STT）和数据持久化
- 剧情数据：Markdown 源文件 → 编译 → 结构化 TypeScript
- 关注分离：故事数据 vs 运行时逻辑
- 轻量化、异步、接口清晰、易于扩展
